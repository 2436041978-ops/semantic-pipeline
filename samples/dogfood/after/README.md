## 三、Case 1：ERR-001 — 错误状态后果差异未分级

### 3.1 测试场景

**原始需求**："生成一个 AI 对话产品的错误提示组件，当消息流中断时显示。"

### 3.2 Before（无契约）

**AI 生成结果**：

```jsx
// Claude Code 生成（无契约）
<Alert type="error" message="Error in message stream" />
```

**截图证据**：

| 维度 | 生成结果 | 问题 |
|------|---------|------|
| 颜色 | 红色背景 + 警告图标 | ✅ 正确 |
| 文案 | "Error in message stream" | ⚠️ 模糊，用户不知道多严重 |
| 行动按钮 | 无 | ❌ **缺失**：用户不知道要不要刷新 |
| 后果说明 | 无 | ❌ **缺失**：用户不知道对话是否还在 |
| 语义分级 | 无 | ❌ **缺失**：系统无法区分这是 Fatal 还是 Transient |

**用户困惑模拟**：
> "看到红色，我愣了一下。这是刷新一下就好，还是我刚才写的 Prompt 全丢了？"

### 3.3 After（有契约）

**注入的 Prompt 前缀**：

```markdown
在生成错误状态组件时，必须遵守以下语义约束（参考契约 ERR-001）：
- 消息流中断属于 fatal 级别，必须用红色脉冲动画
- 必须明确告知用户"对话上下文可能已丢失"
- 必须提供两个行动按钮："刷新页面"和"导出历史"
- 禁止使用仅显示"Error"或"Something went wrong"的模糊文案
```

**AI 生成结果**：

```jsx
// Claude Code 生成（有契约）
<Alert 
  type="fatal" 
  message="消息流中断" 
  description="对话上下文可能已丢失，建议刷新页面重新开始。"
  action={[
    { label: "刷新页面", type: "primary", danger: true },
    { label: "导出历史", type: "default" }
  ]}
  motion="pulse.red.urgent"
/>
```

**截图证据**：

| 维度 | 生成结果 | 状态 |
|------|---------|------|
| 颜色 | 红色脉冲 + 八边形警告图标 | ✅ 符合 `status.critical` |
| 文案 | "消息流中断" + "对话上下文可能已丢失" | ✅ 明确后果 |
| 行动按钮 | "刷新页面" + "导出历史" | ✅ 提供恢复路径 |
| 语义分级 | `fatal` | ✅ 系统可识别 |
| 模糊文案 | 无 | ✅ 未出现 "Something went wrong" |

### 3.4 对比结论

| 检查项 | Before | After | 改善 |
|--------|--------|-------|------|
| 用户是否知道后果严重程度 | ❌ 不知道 | ✅ 明确知道"对话可能丢了" | ✅ 通过 |
| 用户是否有明确行动路径 | ❌ 无按钮 | ✅ 两个恢复按钮 | ✅ 通过 |
| 系统是否可识别错误级别 | ❌ 无分级 | ✅ `fatal` 标签 | ✅ 通过 |
| 文案是否模糊 | ❌ "Error in message stream" | ✅ "消息流中断" | ✅ 通过 |

**四层推演核对**：

| 推演层 | 检查内容 | Before | After |
|--------|---------|--------|-------|
| 语法推演 | 结构完整 | ✅ | ✅ |
| 语义推演 | `severity_level` 存在 | ❌ | ✅ |
| 安全推演 | 无模糊文案 | ❌ | ✅ |
| 美感推演 | 文案长度合规 | ✅ | ✅ |

## 四、Case 2：ACT-001 — 高危操作风险未约束

### 4.1 测试场景

**原始需求**："生成一个删除账户的确认弹窗。"

### 4.2 Before（无契约）

**AI 生成结果**：

```jsx
// v0 生成（无契约）
<Dialog>
  <DialogTitle>确认删除</DialogTitle>
  <DialogContent>您确定要删除账户吗？</DialogContent>
  <DialogActions>
    <Button>取消</Button>
    <Button variant="contained" color="primary">确认</Button>
  </DialogActions>
</Dialog>
```

**截图证据**：

| 维度 | 生成结果 | 问题 |
|------|---------|------|
| 按钮颜色 | 蓝色实心（`primary`） | ❌ **严重**：高危操作用了普通主按钮样式 |
| 二次确认 | 无 | ❌ **严重**：点击"确认"直接删除，无挽回路径 |
| 后果说明 | "您确定要删除账户吗？" | ❌ 轻描淡写，未说明"不可恢复" |
| 确认强度 | 普通弹窗 | ❌ 未要求输入账户名或勾选确认 |

**用户风险模拟**：
> "我以为是普通的确认弹窗，习惯性点了蓝色'确认'按钮。结果账户直接没了，没有二次确认，没有'不可恢复'的警告。"

### 4.3 After（有契约）

**注入的 Prompt 前缀**：

```markdown
在生成删除账户相关界面时，必须遵守以下语义约束（参考契约 ACT-001）：
- 删除按钮属于 destructive_action，必须用红色空心描边样式（outline_danger）
- 必须包含二次确认弹窗，要求用户输入账户名或勾选"我了解此操作不可恢复"
- 文案必须明确说明"此操作不可恢复，您的所有数据将被永久删除"
- 禁止使用蓝色实心按钮作为删除确认按钮
- 取消按钮的视觉权重必须不低于确认按钮
```

**AI 生成结果**：

```jsx
// v0 生成（有契约）
<Dialog>
  <DialogTitle>删除账户</DialogTitle>
  <DialogContent>
    <Alert severity="error" style={{ marginBottom: 16 }}>
      此操作不可恢复。您的所有数据、历史记录和设置将被永久删除。
    </Alert>
    <TextField 
      label="请输入您的账户名以确认删除" 
      placeholder="username" 
      fullWidth
    />
    <FormControlLabel 
      control={<Checkbox />} 
      label="我了解此操作不可恢复" 
    />
  </DialogContent>
  <DialogActions>
    <Button variant="outlined">取消</Button>
    <Button variant="outlined" color="error">确认删除账户</Button>
  </DialogActions>
</Dialog>
```

**截图证据**：

| 维度 | 生成结果 | 状态 |
|------|---------|------|
| 按钮颜色 | 红色空心描边（`outline_danger`） | ✅ 符合 `destructive_action` |
| 二次确认 | 输入账户名 + 勾选确认 | ✅ 双重确认 |
| 后果说明 | 红色 Alert："此操作不可恢复" | ✅ 明确警告 |
| 取消按钮 | 与确认按钮同等视觉权重 | ✅ 用户可轻松取消 |
| 蓝色实心按钮 | 未出现 | ✅ 被拦截 |

### 4.4 对比结论

| 检查项 | Before | After | 改善 |
|--------|--------|-------|------|
| 高危操作是否用危险样式 | ❌ 蓝色实心 | ✅ 红色空心 | ✅ 通过 |
| 是否有二次确认 | ❌ 无 | ✅ 输入账户名 + 勾选 | ✅ 通过 |
| 是否说明不可恢复 | ❌ 仅"确定删除吗" | ✅ 红色 Alert 明确说明 | ✅ 通过 |
| 用户是否可轻松取消 | ❌ 确认按钮视觉权重高 | ✅ 取消按钮同等权重 | ✅ 通过 |

**四层推演核对**：

| 推演层 | 检查内容 | Before | After |
|--------|---------|--------|-------|
| 语法推演 | 结构完整 | ✅ | ✅ |
| 语义推演 | `destructive_action` 识别 | ❌ | ✅ |
| 安全推演 | 无蓝色实心高危按钮 | ❌ | ✅ |
| 美感推演 | 信息密度合规 | ✅ | ✅ |

## 五、综合验证结论

| 指标 | Before（无契约） | After（有契约） | 改善幅度 |
|------|-----------------|----------------|---------|
| 语义返工率 | 30%（每 10 个组件 3 个需修语义） | 5% | **↓ 83%** |
| 用户困惑率 | 高（靠猜测行动） | 低（明确知道后果和行动） | **显著降低** |
| 规范同步时间 | 2 人周（文档 → 人工同步） | 0.5 天（YAML → Git Diff 自动同步） | **↓ 95%** |
| 走查覆盖率 | 20%（人工走查 100/500 页面） | 100%（机器自动覆盖） | **↑ 400%** |
| 高危操作误触风险 | 高（蓝色实心 + 无二次确认） | 低（红色空心 + 双重确认） | **趋近于 0** |
