# 技术设计方案与工程落地规划

> **一句话定义**：当 AI 生成界面时，设计意图在偏离。Schema-As-Code 在语义层建立一套机器可读的约束契约，让 AI 生成的内容在到达视觉层和工程层之前，先过一遍语义审查。不是替代任何工具，是所有工具的上游约束。

**核心架构**：模式库（Pattern Library）+ 契约库（Contract Library）+ 验证工具集（Validation Toolkit）。三阶段不是三个产品，是一条流水线的三个环节：发现问题 → 写契约 → 证明有效。

## 一、技术设计方案

### 1.1 总体架构：确定性编排 + 语义智能体

参考 [paperjury](https://github.com/u7079256/paperjury) 的关键设计原则：

> "确定性编排拥有状态、路由、停止和精确一次补丁应用，而语义智能体仅限于有界审查、判断、起草和审计任务。"

| 层级 | 做什么 | 谁做 | 为什么这样分 |
|------|--------|------|-------------|
| **确定性编排（Deterministic Orchestration）** | 状态管理、路由、停止、精确一次应用 | 硬逻辑（代码写死） | 模式匹配、契约校验、规则归档必须 100% 可靠，不能依赖 AI 随机性 |
| **语义智能体（Semantic Agent）** | 有界审查、判断、起草、审计 | LLM / AI Agent | 需要理解上下文、判断偏差、起草契约，AI 比规则更灵活 |

**映射到 Schema-As-Code**：

- **确定性编排** = 模式库的结构化存储 / 契约库的 YAML 校验 / 不可变边界拦截 / 规则优先级链 / Git 精确一次同步
- **语义智能体** = 组件语义快照的审查 / 偏差判断 / 契约起草 / 验证审计

### 1.2 三阶段流水线架构

```
阶段一：发现问题（Semantic Guard 扫描）
    ↓
阶段二：解决问题（写 YAML 契约）
    ↓
阶段三：证明解决有效（跑验证）
```

**阶段一：发现问题（Semantic Guard 扫描）**

输入组件描述或语义快照，自动匹配已知模式，输出偏差标记和关联契约。

- **输入**：自然语言描述（如"系统报错，红色提示"）或组件语义快照（JSON）
- **处理**：模式库索引匹配（按组件类型 + 关键词）
- **输出**：模式匹配结果（如"匹配到 ERR-001：错误状态后果差异未分级"）+ 证据截图 + 关联契约链接

**阶段二：解决问题（写 YAML 契约）**

读取模式卡片，填写语义规则，生成 YAML 契约文件。

- **输入**：模式卡片（从阶段一导入）+ 设计师填写的语义规则
- **处理**：表单引擎按组件类型动态渲染字段 → YAML 生成器编译
- **输出**：YAML 契约文件 + Prompt 前缀 + 走查清单（可下载 / 可复制）

**阶段三：证明解决有效（跑验证）**

加载 YAML 契约，跑测试用例，输出验证报告。

- **输入**：YAML 契约文件 + 测试用例（组件语义快照集合）
- **处理**：测试引擎（单元/集成/回归）+ 报告生成器
- **输出**：验证报告（通过率曲线 / 拦截率统计 / 断裂点标记 / A/B 对比截图）

## 二、工程设计落地规划

### 2.1 三阶段 MVP 切片

#### 阶段一 MVP：Semantic Guard Web 搜索界面

| 维度 | 内容 |
|------|------|
| **目标** | 让任何人都能输入一个组件描述，自动匹配到已知模式，看到"这个组件可能有什么语义偏差" |
| **形态** | Web 搜索界面（类似搜索引擎，输入即查） |
| **输入** | 自然语言描述（如"系统报错，红色提示"）或组件语义快照（JSON） |
| **输出** | 模式匹配结果（"匹配到 ERR-001：错误状态后果差异未分级"）+ 证据截图 + 关联契约链接 |
| **技术实现** | 前端搜索框 + 模式库索引（按组件类型 + 关键词）+ 结果展示页 |
| **资产沉淀** | 模式库 v0.1（5 个模式卡片）+ 搜索索引 + 组件语义快照模板 |
| **供后期嵌入** | 搜索 API 接口，供 Figma 插件 / Cursor 扩展 / CI 脚本调用 |

#### 阶段二 MVP：契约编辑器（YAML 生成工具）

| 维度 | 内容 |
|------|------|
| **目标** | 让设计师 / DesignOps 不用手写 YAML，通过表单填写生成契约文件 |
| **形态** | Web 表单工具（选择组件类型 → 填写语义属性 → 自动生成 YAML） |
| **输入** | 模式卡片（从阶段一导入）+ 设计师填写的语义规则 |
| **输出** | YAML 契约文件 + Prompt 前缀 + 走查清单（可下载 / 可复制） |
| **技术实现** | 表单引擎（按组件类型动态渲染字段）+ YAML 生成器 + Git 提交集成 |
| **资产沉淀** | 契约库 v0.1（5 个 YAML 文件）+ Prompt 前缀模板库 + 走查清单模板 |
| **供后期嵌入** | YAML 生成 API，供 DesignOps 平台 / Git 仓库自动同步调用 |

#### 阶段三 MVP：验证仪表盘（Validation Dashboard）

| 维度 | 内容 |
|------|------|
| **目标** | 让团队看到"加了契约后，语义错误真的少了" |
| **形态** | Web 仪表盘（输入测试用例 → 自动跑验证 → 展示通过率 / 拦截率 / A/B 对比） |
| **输入** | YAML 契约文件 + 测试用例（组件语义快照集合） |
| **输出** | 验证报告（通过率曲线 / 拦截率统计 / 断裂点标记 / A/B 对比截图） |
| **技术实现** | 测试引擎（单元/集成/回归）+ 报告生成器 + 可视化仪表盘 |
| **资产沉淀** | 验证工具集 v0.1（测试用例库 + 报告模板）+ 历史数据基线 |
| **供后期嵌入** | 验证 API，供 CI 流水线 / 监控平台 / 归因 Dashboard 调用 |

---

### 2.2 三阶段不是三个产品，是一条流水线

```
发现问题 → 写契约 → 证明有效
   ↓          ↓          ↓
Semantic   Contract   Validation
 Guard      Library    Toolkit

不是三个独立产品，是一个工作流的三个环节：
- 阶段一的输出（模式卡片）是阶段二的输入
- 阶段二的输出（YAML 契约）是阶段三的输入
- 阶段三的输出（验证数据）反哺阶段一（更新模式卡片准确率）
```

## 三、组织经济价值 / ROI

### 3.1 语义不一致的隐性成本（以前）

| 成本项 | 以前的做法 | 隐性成本 |
|--------|-----------|---------|
| **前端返工** | AI 生成代码后，前端修语义错误（颜色、文案、按钮样式） | 30% 返工时间，100 个组件 = 2 人天 |
| **规范同步** | DesignOps 发文档、@全员、开会、人工走查 | 2 人周/次，覆盖率 20%，遗漏 80% |
| **用户投诉** | 用户投诉"看不懂报错""按钮点错了"，PM 无法归因 | 每月 20 条投诉，处理成本 2 人天/月 |
| **设计资产流失** | 规范写在语雀/Notion，版本混乱，口头约定 | 每次人员变动，规范重建成本 1 人月 |

### 3.2 有了 Schema-As-Code 以后（以后）

| 维度 | 以前 | 以后 | ROI |
|------|------|------|-----|
| **前端返工率** | 30% | 5% | 语义错误机器预拦截，返工时间 ↓ 83% |
| **规范同步成本** | 2 人周 | 0.5 天 | YAML 契约 Git 管理，Diff 自动同步，成本 ↓ 80% |
| **走查覆盖率** | 20% | 100% | 机器自动审查，人只处理不合格项，覆盖率 ↑ 400% |
| **用户投诉归因** | 无法归因 | 可归因到模式 | 投诉处理时间从 2 天 → 0.5 天 |
| **设计资产沉淀** | 碎片化 | 版本化 | YAML 文件可复用、可迁移，人员变动不影响 |

---

### 3.3 一句话 ROI

> 以前修一个语义错误 10 分钟，100 个错误就是 2 人天。现在写一次 YAML 契约，机器自动拦，返工率从 30% 降到 5%。规范同步从 2 周变成 0.5 天，走查覆盖率从 20% 到 100%。

## 四、开发落地全流程规划

### 4.1 里程碑（16 周）

> **M1-M2 压缩说明**：模式库本质是静态 JSON/YAML 数据 + 前端搜索索引，纯前端实现，无需后端服务。1 个前端工程师 1-2 周可完成 MVP。搜索可用本地 JSON 索引，无需数据库。
>
> **M5-M6 调整说明**：Figma API 和 Cursor 插件生态不稳定，实际开发插件风险高、周期长。改为输出"嵌入资产文档"（设计稿/接口规范/脚本模板），供后续团队参考，而非直接开发插件。

| 里程碑 | 时间 | 目标 | 交付件 | 可行性 | 风险 | 替补方案 |
|--------|------|------|--------|--------|------|---------|
| **M1** | 第 1-2 周 | 模式库 v0.1 | 5 个模式卡片 JSON + 搜索索引 | 高 | 数据结构不稳定（v1.0 冻结 8 字段） | Notion/语雀数据库手动查询 |
| **M2** | 第 3-4 周 | Semantic Guard Web 搜索界面 | 可访问的搜索界面 + REST API | 高 | 同 M1 | 静态网页 + 本地 JSON |
| **M3** | 第 5-8 周 | 契约编辑器 MVP | 表单生成 YAML 工具 + 5 个契约文件 + Prompt 前缀模板 | 中 | YAML 结构兼容性（v1.0 冻结字段名） | VS Code 插件 + YAML Snippet |
| **M4** | 第 9-12 周 | 验证仪表盘 MVP | 测试用例库 × 10 + 报告工具 + 历史基线 | 中 | 测试用例覆盖不全 | 手动跑测试 + 人工复核 |
| **M5** | 第 13-14 周 | 全链路闭环 Demo | 搜索 → 生成契约 → 跑验证 完整流程演示 | 中 | 三阶段接口对接 | 分阶段演示，非一键自动化 |
| **M6** | 第 15-16 周 | 消费方嵌入资产文档 | Figma 插件设计稿 / Cursor 扩展接口规范 / CI 脚本模板 | 高（文档而非代码） | 无（不开发实际插件） | 无 |

### 4.2 技术可行性详细评估

#### M1-M2：模式库 + Semantic Guard（可行性：高）

| 评估项 | 结论 | 依据 |
|--------|------|------|
| **技术复杂度** | 低 | 纯前端，HTML/CSS/JS 即可，无需框架 |
| **数据存储** | 本地 JSON 文件 | 5 个模式卡片 < 100KB，无需数据库 |
| **搜索实现** | 前端字符串匹配 | 数据量小，无需 Elasticsearch/Algolia |
| **人力成本** | 1 个前端工程师 × 2 周 | 静态页面 + 搜索框 + 结果展示 |
| **风险点** | 数据结构变动 | 对策：v1.0 冻结模式卡片 8 字段结构，后续只增不减 |
| **替补方案** | Notion/语雀数据库 | 如果 Web 界面来不及，先用文档工具手动匹配 |

#### M3-M4：契约编辑器 + 验证仪表盘（可行性：中）

| 评估项 | 结论 | 依据 |
|--------|------|------|
| **技术复杂度** | 中 | 表单引擎需按组件类型动态渲染字段（Alert 和 Button 表单不同） |
| **YAML 生成** | 需处理嵌套结构 | 使用 js-yaml 库，前端生成即可 |
| **结构校验** | 硬逻辑校验 | 校验必填字段：intent_id / semantic_tokens / immutable_boundaries / llm_constraints |
| **人力成本** | 1 个全栈工程师 × 4 周 | 前端表单 + 后端 YAML 生成 + 结构校验 |
| **风险点** | YAML 结构兼容性 | 对策：v1.0 冻结 YAML 字段名，只增字段不改字段名 |
| **替补方案** | VS Code 插件 + Snippet | 如果 Web 编辑器来不及，用 IDE 模板手动生成 YAML |

#### M5-M6：全链路闭环 + 消费方嵌入（可行性：低 → 调整为文档输出）

| 评估项 | 原方案（开发插件） | 调整后方案（输出文档） |
|--------|------------------|----------------------|
| **Figma 插件** | 需学习 Figma Plugin API，API 不稳定，调试周期长 | 输出插件设计稿 + 接口规范文档 |
| **Cursor 扩展** | Cursor 插件生态不成熟，文档缺失 | 输出扩展接口规范 + 接入指南 |
| **CI 脚本** | 需对接 GitHub Actions/GitLab CI，各平台语法不同 | 输出脚本模板（GitHub/GitLab 双版本） |
| **人力成本** | 2 个前端工程师 × 4 周 | 1 个技术写作者 × 2 周 |
| **风险** | 高（API 变动导致重写） | 低（文档无技术债务） |
| **价值** | 高（但实际落地难） | 中（为后续团队提供参考） |

**调整决策**：M5-M6 不开发实际插件，改为输出"嵌入资产文档包"。理由是：

1. Figma/Cursor 插件 API 不稳定，16 周开发周期内可能遭遇 Breaking Change
2. 插件需要设计师/前端深度配合，协调成本高
3. 文档资产可立即被消费方参考，技术债务为零

### 4.3 角色协同：语义翻译者在全流程中的位置

你不是替代设计师/前端/DesignOps/PM，而是在他们之间建立一层机器可读的语义契约。

```
PM 的意图 → 设计师的语义定义 → YAML 契约 → 前端的 Prompt 前缀 → AI 生成 → 验证审计
    ↑                                                              ↓
    └──────────── 验证数据反哺 ─────────────────────────────────────┘
```

---

### 4.4 上下游输入输出件

| 角色 | 输入（给 Schema-As-Code） | 输出（从 Schema-As-Code 获得） |
|------|--------------------------|------------------------------|
| **设计师** | 设计意图描述（自然语言）+ 组件截图 | 模式匹配建议（"这是 ACT-001 类型"）+ 契约草稿 |
| **前端** | 组件代码（React/Vue Props）+ AI 生成输出 | Prompt 前缀（贴在 AI 指令前）+ CI 校验报告 |
| **DesignOps** | 规范变更需求（自然语言） | YAML 契约文件 + Git 同步脚本 + 影响面分析 |
| **AI PM** | 用户投诉文本 + 原型描述 | 风险清单（生成前检查）+ 归因报告（投诉后分析） |
| **数据工程师** | 数据字段定义变更 | 字段 ↔ 语义令牌 ↔ 组件 Props 三向映射表 |
| **业务分析师** | 业务规则调整 | 规则变更影响面分析 + 组件呈现同步清单 |
| **算法工程师** | 模型版本更新 + 标签阈值调整 | 模型标签 ↔ 组件语义映射校验报告 |

### 4.5 交付件与交付标准

#### 阶段一交付件：模式库 v0.1

| 交付件 | 标准 | 验收方式 | 替补方案 |
|--------|------|---------|---------|
| 模式卡片 × 5 | 每个卡片包含：模式 ID / 组件类型 / 组件手册依据 / 症状 / 根因 / 产品实例 / 关联契约 / 验证方法 | 设计师能看懂，前端能定位到组件手册 | Notion/语雀数据库 |
| 搜索索引 | 支持按组件类型 + 关键词检索 | 输入"删除按钮"，3 秒内返回 ACT-001 | 本地 JSON 文件 |
| Web 搜索界面 | 输入框 + 结果展示页 + 模式详情页 | 任何人无需登录即可使用 | 静态网页 |
| 搜索 API | RESTful 接口，返回 JSON | 前端能调用，返回格式统一 | 本地 JSON 读取 |

#### 阶段二交付件：契约库 v0.1

| 交付件 | 标准 | 验收方式 | 替补方案 |
|--------|------|---------|---------|
| YAML 契约 × 5 | 结构完整：intent_id / semantic_tokens / immutable_boundaries / llm_constraints | 通过 YAML 结构校验（无缺失字段） | VS Code Snippet |

## 五、技术背书

### 5.1 阿里云：约束基建（Constraint Infrastructure）

阿里云在《构建可审计、可进化的 AI Agent 底座》中提出：

> "约束基建是 AI Agent 的可信底座。没有约束的 Agent 是失控的，约束必须可审计、可进化。"

- **可审计** → 验证工具集（机器自动走查，留下审计痕迹）
- **可进化** → 契约库的版本管理（Git Diff，变更可追溯）
- **约束基建** → 模式库 + 契约库 + 验证工具集

阿里云做后端 Agent 的约束基建，Schema-As-Code 做前端界面的约束基建。两者叠加，才是完整的 AI 约束体系。

---

### 5.2 paperjury：确定性编排 vs 语义智能体

paperjury 的关键设计原则：

> "确定性编排拥有状态、路由、停止和精确一次补丁应用，而语义智能体仅限于有界审查、判断、起草和审计任务。"

**Schema-As-Code 的映射**：

- **确定性编排** = 模式库的结构化存储 / 契约库的 YAML 校验 / 不可变边界拦截 / 规则优先级链 / Git 精确一次同步
- **语义智能体** = 组件语义快照的审查 / 偏差判断 / 契约起草 / 验证审计
